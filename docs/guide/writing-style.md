# 正文写作与排版规范

<p class="lead">这套规范适用于站内所有以 Markdown 编写的正文，包括文章、文档与项目说明。格式负责说明内容的结构和语义，具体的颜色、间距与响应式效果由站点主题统一处理。</p>

::: tip 使用原则
先选择语义正确的格式，再考虑视觉效果。若删掉样式后仍能看懂层级、重点和关系，通常说明格式用对了。
:::

## 页面起点

每页先写 Frontmatter，再写一个且仅一个一级标题。`title` 用于浏览器标题、搜索和分享卡片；`description` 应用一句完整的话说明页面内容。

```md
---
title: 页面标题
description: 一句话说明本页的主要内容。
---

# 页面标题

用一小段导语交代本文讨论什么，以及读者为什么需要继续读。
```

- Frontmatter 中的 `title` 与正文一级标题应保持一致，允许为了行文自然做轻微措辞调整。
- 页面语言不是简体中文时，增加 `lang`，例如 `lang: en` 或 `lang: ja`。
- 不用粗体段落模拟标题，也不为了字号跳过标题层级。

## 标题与段落

二级标题划分文章的大部分，三级标题拆分其中的问题或步骤。四至六级标题只在内容确实复杂时使用；若连续出现很多低层级标题，通常应先重组文章。

```md
## 大节标题

这一节先给出完整的上下文。

### 小节标题

围绕一个更具体的问题展开。

#### 必要时使用的更低层级

不要从二级标题直接跳到四级标题。
```

<p class="format-demo-label">展示效果</p>

<h3>三级标题：一个具体问题</h3>
<p>正文自然接在标题之后。一个段落集中表达一个中心意思，段落之间留一个空行。</p>

<h4>四级标题：必要的细分</h4>
<p>低层级标题仍然应能独立概括其后的内容，而不是只承担装饰作用。</p>

<h5>五级标题：较少使用</h5>
<p>只有长篇技术文档或层次复杂的资料通常会需要这一层。</p>

<h6>六级标题：最后一级</h6>
<p>若还需要继续细分，应考虑拆成新页面。</p>

普通正文直接连续书写，由浏览器根据阅读宽度自动换行。只有诗歌、地址或逐行对应的文本才使用 `<br>` 强制换行，不要用连续空格或多个空行制造间距。

## 行内文本

行内格式只标记局部语义，不应让整段文字同时出现多种强调。

```md
这是普通正文，**粗体给出关键结论**，*斜体标记术语或语气*，
~~删除线表示已明确废弃的内容~~，<mark>高亮标出需要回看的词句</mark>。

配置项写作 `site-reading`，快捷键写作 <kbd>Ctrl</kbd> + <kbd>K</kbd>。
<abbr title="HyperText Markup Language">HTML</abbr> 首次出现时可给出全称；
H<sub>2</sub>O 与 10<sup>2</sup> 只在确有上下标含义时使用。
```

- **粗体**用于结论、风险或必须注意的对象，不用于整段加粗。
- *斜体*用于术语、作品名补充或轻微语气强调，中文正文中应克制使用。
- <mark>高亮</mark>表示“需要回看”，不能代替标题或粗体。
- `行内代码`只包裹命令、文件名、参数、字段和值，不包裹普通中文概念。
- ~~删除线~~保留修订痕迹；正式改写且无需保留旧说法时，直接删除原文。

## 链接

链接文字应单独读起来也能说明去向，避免连续使用“这里”“点击查看”。

```md
先阅读[维护这个网站](/guide/getting-started)，再执行发布前检查。
```

站内页面使用以 `/` 开头的绝对路径；外部资料使用完整网址。若链接用于下载、打开新窗口或跳转到站外，应在上下文中提前说明。

## 列表

无序列表表示并列项目，有序列表表示顺序或优先级。每个列表项应保持相近的语法结构。

```md
- 先写结论
- 再给证据
  - 只在确有从属关系时嵌套
  - 嵌套层级尽量不超过两层
- 最后说明限制

1. 保存修改
2. 运行检查
3. 发布并验证
```

短语式列表可以不加句号；只要任一项是完整句子或包含多个分句，所有列表项都应使用一致的句末标点。

## 引用与出处

短引用可直接放在正文中并加引号；需要独立呈现时使用引用块。引用不是普通段落的装饰，能找到来源时应同时给出作者、作品或链接。

```html
<blockquote>
  <p>好的格式不会抢走内容的声音，只会让它更容易被听见。</p>
  <footer>— <cite>本站写作规范</cite></footer>
</blockquote>
```

引用其他语言时，在容器或段落上写明语言，例如 `<blockquote lang="en">`。不要只写“网传”或无法核验的二手署名。

## 代码

短命令、参数和文件名使用行内代码。多行代码使用围栏代码块，并写明语言，以获得语法高亮和更准确的复制内容。

- 命令与输出分开呈现，并为不同语言分别建立代码块。

## 表格

表格适合比较字段明确、列数稳定的信息。叙述、步骤或长段落应改用正文与列表；列较多时还要检查窄屏横向滚动是否可用。

```md
| 格式 | 适合表达 | 不适合表达 |
| --- | --- | --- |
| 列表 | 并列项或步骤 | 大段论证 |
| 表格 | 多项对照 | 阅读顺序 |
| 引用 | 保留原话 | 普通提示 |
```

## 术语与定义

连续解释多个术语时，可以使用语义化的定义列表。它需要少量 HTML，但比“加粗词语 + 普通段落”更准确。

```html
<dl>
  <dt>正文</dt>
  <dd>承担主要叙述、论证或说明的内容。</dd>
  <dt>元数据</dt>
  <dd>用于描述页面标题、摘要和语言等信息的数据。</dd>
</dl>
```

## 提示块

提示块用于把正文之外的补充、建议、风险和严重警告分级。标题应说明具体内容，不要让整篇文章被彩色块切碎。

```md
::: info 背景
补充理解正文所需的上下文。
:::

::: tip 建议
给出更省力或更稳妥的做法。
:::

::: warning 注意
说明容易踩坑但仍可恢复的问题。
:::

::: danger 风险
说明会造成数据丢失、安全问题或不可逆后果的操作。
:::
```

## 折叠内容与分隔线

补充材料会打断主线、但读者可能仍需查看时，使用折叠内容。分隔线只表示明显的场景转换，不替代标题。

```md
::: details 查看补充说明
这里放背景资料、完整输出或次要示例。
:::
```

## 图片、图注与无障碍文本

图片不是文本格式，但属于常见正文内容。图片必须有能替代其信息的 `alt` 文本；只有纯装饰图片才使用空的 `alt=""`。图注解释“读者应看什么”，不要重复图片文件名。

```html
<figure>
  <img src="/images/example.webp" alt="浅色与深色正文排版的并排对比">
  <figcaption>同一组正文格式在两种主题下保持一致的层级。</figcaption>
</figure>
```

大图应先压缩到合适尺寸；截图中的关键文字还要在正文中复述，不能让信息只存在于图片里。

## 发布前检查

- 页面只有一个一级标题，且标题层级没有跳跃。
- 段落、列表和表格各自承担合适的信息关系。
- 粗体、高亮、提示块没有多到互相争夺注意力。
- 链接文字能说明去向，引用能说明来源。
- 代码块标明语言，图片具有有效替代文本。
- 在桌面与手机宽度、浅色与深色主题下都能顺畅阅读。
